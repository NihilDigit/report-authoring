# 行文风格

本文针对**正文内容**（阶段 2 md 成稿与阶段 4 回填 docx 时的文字本身），不涉及排版。

## 目标

**平实、内容丰富的中文书面语**。报告是学生/作者呈给老师/评审的正式文本，不是技术博客、不是
commit log、不是聊天。

## 硬性避免（6 条，不可商量）

这六类问题出现一次就算文字不合格。写完一段先自查，再交给用户。

### 1. 口语

```
❌ 你看这个 Protobuf 啊，它就是个序列化工具，挺方便的。
❌ 搞了半天才跑通，环境坑挺多。
❌ 反正最后结果是对的。
```

**为什么不行**：报告用的是书面语，"你看"、"挺"、"搞"、"反正"属于口语代词和语气词，
降低正式度，显得随意。

```
✅ Protocol Buffers 是一种紧凑的二进制序列化格式，常用于跨语言数据交换。
✅ 环境配置过程中出现若干依赖版本冲突，本节逐项说明解决方法。
```

### 2. 大厂/行业黑话

```
❌ 本项目打通了数据采集与存储的全流程闭环，实现了端到端的序列化能力。
❌ 我们对齐了各 Part 的交付路径，最后赋能了整份报告的产出。
❌ 这套 pipeline 具备高可用性和强扩展性。
```

**为什么不行**："打通"、"闭环"、"链路"、"端到端"、"赋能"、"对齐"、"抓手"、"颗粒度"等词
在互联网公司内部泛滥，放到学术报告里显得浮夸空洞，且本身不承载具体信息。

```
✅ 本实验按要求顺序完成了序列化、数据传输、气象数据采集三部分内容。
✅ 气象站客户端将采集数据编码后，通过 UDP 发送到服务器，服务器接收后以 Avro 格式存储。
```

### 3. 过度工程化 / 细节过度

```
❌ 由于 protoc 34.1 生成的 Java 代码调用了新版 protobuf-java 运行库中的 Descriptor API
   （具体为 getMessageType(int) 方法），而默认的 pom.xml 配置使用了 4.29.3 版本，
   因此产生 NoSuchMethodError 异常，需要将 protobuf-java 升级到 4.34.1 以对齐。
```

**为什么不行**：正文不是 commit log / issue 复盘。`getMessageType(int)` 这种 API 细节
属于实现细节，只在"结果分析 / 遇到的问题"这类小节以简短一句交代原因即可。

```
✅ 初次编译时出现 protobuf-java 与 protoc 版本不匹配的错误，将 protobuf-java 升级到 4.34.1
   后解决。
```

### 4. 汉语欧化

典型表现：名词化句式（"对…的使用"）、多余介词、长定语从句堆叠。

```
❌ 通过本次实验的进行，我们完成了对 Protocol Buffers 的使用以及相应功能的实现。
❌ 该方案提供了一个针对跨语言数据交换问题的解决方式。
❌ 在对于代码进行编译的过程中，由于版本的不匹配而导致了错误的产生。
```

**为什么不行**：
- "通过…的进行"、"对…的使用"——把动词名词化再套介词，啰嗦
- "提供了一个…的解决方式" = "解决了"
- "由于 X 而导致 Y 的产生" = "X 导致 Y"

```
✅ 本次实验使用 Protocol Buffers 实现了相应功能。
✅ 该方案解决了跨语言数据交换问题。
✅ 编译时因版本不匹配报错。
```

### 5. 英语思维翻译腔

```
❌ 本实验基于 Python 3.13 进行了 Protobuf 相关功能的实现。
❌ 通过使用 UDP 协议，我们实现了从客户端到服务器的数据的传输。
❌ 代码被执行之后，相应的输出将被打印到终端。
```

**为什么不行**：
- "基于…进行…的实现" = based on X, do Y implementation（中文直接说"用 X 做 Y"）
- "通过使用" = by using（多余，直接"用"）
- 滥用被动 "被执行之后" 、"将被打印" —— 英文被动语态直译

```
✅ 本实验用 Python 3.13 实现了 Protobuf 序列化功能。
✅ 客户端通过 UDP 将数据发送到服务器。
✅ 运行代码后，终端会打印输出。
```

### 6. Overclaim 或加太多限制条件

```
❌ Protocol Buffers 是最高效的跨语言序列化方式。
❌ 本实验完美展示了大数据采集与传输的全部核心技术。
❌ 在理想网络条件下，排除 GC 抖动、仅计算纯编码时间、忽略 socket 建立开销的前提下，
   Protobuf 的速度大约是 JSON 的 3 倍。
```

**为什么不行**：
- "最高效"、"完美"、"全部"——绝对词，一加就站不住
- 第三条相反方向的问题：限制条件堆到句子自己都动不了，读者看完只记得前提而忘了结论

```
✅ Protocol Buffers 是一种紧凑高效的跨语言序列化方式，本实验使用它完成数据结构的编码。
✅ 本实验覆盖了大数据采集与传输的基本技术要点。
✅ 实测中 Protobuf 编码后的体积约为等价 JSON 的 40%，解析速度也更快。
```

## 三种可选风格（阶段 0 需求对齐时让用户选）

硬性 6 条之外，**"书面语"本身的松紧程度**有差异。写第一段前给用户看三段样例，
确认偏好再继续。

### 风格 A · 严肃学术（论文本科/研究生课程设计）

长句偏多，结构连接词完整（首先/其次/综上所述），适度使用被动。

```
本实验基于 Protocol Buffers 实现结构化数据的序列化与反序列化。首先，根据实验要求，
设计包含嵌套 message 和 repeated 字段的 Book 数据结构；其次，利用 protoc 编译器分别
生成 Python 与 Java 的数据访问类；最后，在两种语言中构造等价的 Book 实例并完成序列化，
所得二进制字节数一致，表明跨语言数据交换的正确性得到了验证。
```

### 风格 B · 平实书面（大多数实验报告）★ 推荐默认

陈述清楚即可，不堆结构词，不炫技，不口语。

```
本实验用 Protocol Buffers 对结构化数据做序列化与反序列化。按实验要求，设计了含嵌套
message 和 repeated 字段的 Book 结构，再用 protoc 分别生成 Python 和 Java 访问类。
两种语言各自构造相同的 Book 对象并序列化，结果均为 154 字节，字节级一致。
```

### 风格 C · 技术文档（简短工程类汇报）

短句为主，要点式，省略不必要的主语，技术术语直接用原文。

```
用 Protocol Buffers 做数据序列化。Book 结构含嵌套 message、repeated 字段。
protoc 编译生成 Python、Java 两份访问类。两语言各自序列化 Book 得 154 字节，
结果一致。
```

## 阶段 0 对齐清单

写正文前给用户过一遍：

- [ ] **风格**：A 严肃学术 / B 平实书面 / C 技术文档（或明示混用）
- [ ] **人称**：第一人称复数（"我们"）/ 第一人称单数（"本人 / 我"）/ 无人称（"本实验" / 被动式）
- [ ] **时态感**：过去式陈述（"完成了"）/ 现在式描述（"使用"）—— 中文无严格时态，但语感不同
- [ ] **术语处理**：英文原词（Protocol Buffers）/ 中文译名（协议缓冲区）/ 混用（首次出现时注明）
- [ ] **评价尺度**：可说"有效 / 常用 / 合理"，避免"最 / 完美 / 全面"——见硬性第 6 条

## 自查清单（写完每段后过一遍）

- 有没有"你看 / 挺 / 搞 / 反正"之类口语词？
- 有没有"闭环 / 链路 / 赋能 / 打通"之类黑话？
- 有没有 pom.xml 里某个具体 API、某条 commit 的细节？搬到脚注/附录更合适？
- 有没有"通过…的进行"、"对…的使用"、"由于 X 而导致 Y 的产生"这类欧化？
- 有没有"基于…进行…"、"被…"、"通过使用…"这类翻译腔？
- 有没有"最 / 完美 / 全部 / 所有 / 一定"这类绝对词？或者前置条件层层叠叠把结论湮没？

## 对 Agent 的元指导

- **风格对齐放在阶段 0**：不要等写完半篇再让用户反馈，成本太高
- **给三段样例、不要空问**："你想要什么风格" 用户没法答。给 A/B/C 三段样例问"哪个接近"
- **硬性 6 条写完自查**：提交一段给用户前，自己过一遍自查清单
- **不怕删**：如果某句话既空洞又堆了限制条件，整句删掉比改更快
- **术语 / 专名不翻译**：protoc、Maven、Thrift 直接用英文，不要译作"协议编译器"、"马文"
